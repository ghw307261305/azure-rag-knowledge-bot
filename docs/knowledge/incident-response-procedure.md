# 障害対応手順

## 概要
本番環境で障害が発生した際の検知・エスカレーション・対応・クローズまでの手順を定義します。

## 障害レベル定義

| レベル | 名称 | 基準 | 目標復旧時間（RTO） |
|-------|------|------|-----------------|
| P1 | 重大 | サービス全停止 / 全ユーザーに影響 / データ損失のリスク | 2時間以内 |
| P2 | 高 | 主要機能停止 / 一部ユーザーに重大影響 | 4時間以内 |
| P3 | 中 | 一部機能劣化 / 回避策あり | 翌営業日内 |
| P4 | 低 | 軽微な表示不具合 / 業務継続可能 | 1週間以内 |

## 検知方法

1. **監視アラート**: Azure Monitor / Application Insights からの自動通知
2. **ユーザー報告**: サポート窓口へのお問い合わせ
3. **定期監視**: 毎時0分・30分に自動ヘルスチェック

## 初動対応（15分以内）

1. 状況確認
   - Azure Portal でリソース状態確認
   - Application Insights でエラーレート確認
   - ログストリームでエラー内容確認
2. 影響範囲の特定
   - 影響ユーザー数・機能を明確化
   - 障害レベルを暫定判定
3. 初報を関係者に送信
   - 送信先: チームSlack #障害対応 チャンネル
   - 内容: 発生日時・症状・暫定レベル・対応者

## エスカレーション先

| レベル | エスカレーション先 | 連絡方法 |
|-------|----------------|---------|
| P1 | リーダー・PMO・クライアント担当 | 電話 + Slack |
| P2 | リーダー・PM | Slack + メール |
| P3 | チーム内 | Slack |
| P4 | チケット登録のみ | Jira/GitHub |

## よくある障害パターンと対処

### APIがタイムアウトする
1. App Serviceのスケールアウト状況を確認
2. Azure OpenAI のレート制限を確認（Portal > 診断 > スロットリング）
3. Azure AI Search のインデックス状態を確認
4. 必要に応じてApp Serviceを再起動（影響を事前告知）

### ログインできないユーザーが多数発生
1. 認証サービス（Entra ID）のステータスページを確認
2. キャッシュサーバーの状態確認
3. 該当ユーザーのアカウント状態をDBで直接確認

### バッチ処理が失敗した
1. バッチログを確認（storage account または Application Insights）
2. 依存サービスの状態確認
3. 手動再実行 or スキップの判断
4. データ整合性への影響を確認

## 復旧後の対応

1. 正常稼働の確認（ヘルスチェック・主要機能テスト）
2. 影響ユーザーへの事後通知（P1/P2のみ）
3. 事後報告書の作成（P1/P2は翌営業日、P3は1週間以内）
4. 再発防止策の検討・実施
