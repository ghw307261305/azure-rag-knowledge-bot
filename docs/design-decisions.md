# 設計判断記録（Design Decisions）

本ドキュメントでは、各設計上の判断とその理由を記録します。
「なぜそうしたか」を説明できることが中高級エンジニアとしての必須要件です。

---

## 1. RAG アーキテクチャ: classic RAG を選択

**判断**: agentic retrieval ではなく classic RAG を採用した。

**理由**:
- 7日間 POC のスコープでは、シンプルで制御しやすい構成が優先
- agentic retrieval は GA 前の機能も多く、ドキュメントが変動しやすい
- classic RAG のほうが「検索→コンテキスト→生成」の流れを面接で説明しやすい
- 将来的に agentic retrieval に移行する際の変更点も明確にしやすい

**トレードオフ**:
- 複雑な多段検索や動的プランニングは対応できない
- 代わりに query rewrite でシングルパスの精度を補う

---

## 2. 検索方式: Hybrid Search（keyword + vector）

**判断**: 純粋なベクトル検索ではなくハイブリッド検索を採用した。

**理由**:
- キーワード検索は固有名詞・型番・略語など厳密一致に強い
- ベクトル検索は意味的類似性に強いが、キーワードの完全一致を見逃すことがある
- Azure AI Search の RRF（Reciprocal Rank Fusion）が両者の結果を自動マージする
- 業務文書では「スカウト」「P1障害」のような専門用語が多く、keyword が有効

**トレードオフ**:
- 純粋ベクトル検索より処理が若干重くなる
- スコアが 0〜1 ではなく RRF スコアになるため解釈に注意が必要

---

## 3. チャンク設計: Markdown 見出し単位で分割

**判断**: 固定トークン数ではなく Markdown の `##` 見出し単位で分割した。

**理由**:
- `##` 見出しが意味的なセクション境界を表すため、内容の一貫性が保たれる
- 固定 500 トークン分割では文脈が途切れるリスクがある
- 各チャンクに title と section を付与することで citation が明確になる

**トレードオフ**:
- セクションが長い文書は 1 チャンクが大きくなる可能性がある
- 見出しのない文書は「概要」セクションとして一括処理される

---

## 4. Embedding モデル: text-embedding-3-large（3072次元）

**判断**: text-embedding-ada-002（1536次元）ではなく text-embedding-3-large を採用した。

**理由**:
- text-embedding-3-large は ada-002 より精度が高い（MTEB ベンチマーク参照）
- 日本語テキストの意味的類似性の捕捉精度が向上している
- コストは ada-002 と同等レベル

**トレードオフ**:
- ベクトルの次元数が 3072 になるためインデックスのストレージ使用量が増加
- Embedding 生成に若干の時間がかかる

---

## 5. Query Rewrite: GPT-4o で検索クエリを書き換え

**判断**: ユーザーの自然文を検索に最適なクエリに書き換えてから検索する。

**理由**:
- 「〜はどうすればいいですか」のような語尾は検索には不要なノイズ
- 重要キーワードを抽出することでベクトル検索の精度が向上する
- 書き換えが失敗しても元の質問でフォールバックするため安全

**トレードオフ**:
- API 呼び出しが 1 回増えるため latency が増加（平均 +1〜2 秒）
- トークン消費量も増える

---

## 6. フォールバック設計: スコア閾値で制御

**判断**: 検索スコアが閾値未満の場合は固定文言を返す。

**理由**:
- 関連文書がない質問に対して GPT が回答を「作り上げる」（hallucination）を防ぐ
- 「わかりません」を明示することのほうが誤情報を返すよりはるかに安全

**現在の設定**:
- `MIN_SCORE_THRESHOLD = 0.01`（Azure AI Search の RRF スコアは低めのため）

**トレードオフ**:
- 閾値が低すぎると hallucination リスクが残る
- 閾値が高すぎると正当な質問もフォールバックになる
- 本番では実際のクエリログを分析して閾値を調整する必要がある

---

## 7. 認証設計: ローカルは API Key、Azure 上は Managed Identity

**判断**: 環境によって認証方式を切り替える設計にした。

**理由**:
- API Key をコードにハードコードするのはセキュリティリスク
- Azure 上では Managed Identity を使うことでキーの管理が不要になる
- DefaultAzureCredential により環境を意識せず認証できる

**本番移行時の対応**:
1. App Service のシステム割り当て Managed Identity を有効化
2. Azure OpenAI と AI Search に適切な RBAC ロールを付与
3. コード側の AzureKeyCredential を DefaultAzureCredential に切り替え

---

## 8. Citation 重複排除: ソースファイル単位

**判断**: 同一ファイルから複数チャンクが取得された場合、引用は 1 件にまとめる。

**理由**:
- 同じドキュメントの異なるセクションが複数ヒットしても、ユーザー体験的には 1 つの文書として扱うほうが自然
- citations の数が膨れ上がることで視認性が低下するのを防ぐ

**トレードオフ**:
- 同じドキュメントの複数セクションが回答に使われていても、表示は 1 件になる
- どのセクションが使われたかを詳しく見るには retrieved_chunks を参照する必要がある
