内容分为两部分：

- 日语简历项目描述
- 面试时可直接使用的中高级话术问答稿

## 1. 日语简历项目描述

### 1.1 职务经历史短版

```text
案件名：Azure RAG Chatbot PoC基盤の設計・実装
役割：技術検証、アプリ基盤設計、フロント/バックエンド実装
期間：自己研鑽プロジェクト
概要：
Azure OpenAI / Azure AI Search を前提とした社内ナレッジ向け RAG チャットボットの PoC 基盤を、local-first で設計・実装。FastAPI と React/TypeScript を用いて、API 契約、画面、引用表示、設定管理、レビュー観点、ドキュメント整備まで一通り対応。
担当内容：
・FastAPI による API 基盤の実装（health/chat エンドポイント、レスポンスモデル定義）
・React/TypeScript によるチャット UI 実装（質問入力、回答表示、引用表示、エラー/ローディング対応）
・環境変数設計、ディレクトリ構成整理、将来の Azure 接続を見据えた責務分離
・アーキテクチャ説明書、API 仕様書、レビュー観点一覧、ナレッジ文書サンプルの作成
成果：
・PoC 段階でも継続開発しやすいフロント/バックエンド分離構成を整理
・API 契約とドキュメントを先行整備し、後続の Azure 接続や検索実装の差し替えを容易化
・モックベースで UI/API の動作確認とテストを先行実施し、実装初期の手戻りを抑制
技術：
Python, FastAPI, Pydantic, TypeScript, React, Vite, Git
Azure OpenAI / Azure AI Search（接続前提の構成設計）
```

### 1.2 职务经历史详版

```text
Azure OpenAI と Azure AI Search を利用することを前提に、社内ナレッジ向け RAG チャットボットの PoC 基盤を self-driven で設計・実装しました。初期段階では local-first の方針を取り、いきなり Azure 実接続へ進むのではなく、FastAPI による API 契約定義、React/TypeScript による画面実装、引用表示、設定管理、レビュー観点整理、知識文書整備を先行させました。

バックエンドでは、FastAPI を用いて health/chat エンドポイントを実装し、Pydantic による入出力モデルを定義しました。将来的な Azure AI Search の検索処理や Azure OpenAI の回答生成を差し替えやすいよう、サービス層を分離した構成にしています。フロントエンドでは、React/TypeScript でチャット UI を作成し、質問入力、回答表示、引用元表示、検索デバッグ表示、ローディングおよびエラー状態のハンドリングを実装しました。

また、単なる動作確認で終わらせず、アーキテクチャ説明書、API 仕様書、レビュー観点一覧、既知課題一覧、ナレッジ文書サンプルを整備しました。PoC であっても、後続の Azure 接続、検索精度改善、CI/CD 追加に発展しやすい形で基盤を整理した点がこの取り組みのポイントです。
```

## 2. 面试时的中高级话术问答稿

### Q1. この案件で何を担当しましたか。

**A.**  
PoC 基盤の全体設計と実装を担当しました。具体的には、FastAPI による API 基盤、React/TypeScript によるチャット UI、設定管理、ドキュメント整備、レビュー観点整理までを一通り対応しています。

### Q2. なぜ最初から Azure 実接続をせず、local-first にしたのですか。

**A.**  
PoC 初期で重要なのは、まず UI、API 契約、責務分離、知識文書の構造を固めることだと判断したためです。先にそこを固めることで、後から Azure OpenAI や Azure AI Search を接続しても手戻りが少なくなります。

### Q3. この構成で意識した設計ポイントは何ですか。

**A.**  
フロントとバックを分離し、バックエンド側で設定管理と回答ロジックを吸収する構成にしました。将来的に検索、Embedding、回答生成を差し替えても API 契約を大きく変えずに済むようにした点を重視しています。

### Q4. 中級以上らしい工夫はどこにありますか。

**A.**  
動かすだけでなく、API 仕様書、アーキテクチャ説明、レビュー観点、既知課題を先に整備した点です。PoC でも後続の人が読み替えやすい状態を作るのは、中級以上で求められる観点だと考えています。

### Q5. コードレビューでは何を見ますか。

**A.**  
責務分離、設定の外出し、例外処理、型定義、将来の差し替え容易性を見ます。今回は特に、Azure 接続ロジックを UI に漏らさないこと、シークレットをコードへ埋め込まないこと、モックから実装へ移行しやすい構造を意識しました。

### Q6. Azure 実案件経験はどの程度ありますか。

**A.**  
本番案件としてはこれからですが、Azure OpenAI / Azure AI Search を前提とした PoC 基盤を自走で設計・実装し、必要な構成、API、ドキュメント、レビュー観点までは整理済みです。Salesforce と AI 開発で培った設計、API 連携、品質担保の経験はそのまま活かせるため、キャッチアップは速いと考えています。

### Q7. Salesforce 経験はこの案件でどう活かせますか。

**A.**  
Salesforce での設計、権限管理、データ連携、バッチ、運用設計の経験は、Azure 案件でもそのまま再利用できます。クラウドサービスは変わっても、設計書作成、仕様整理、レビュー、障害時の切り分けといった基本は共通です。

### Q8. RAG の実装で今後追加するとしたら何を優先しますか。

**A.**  
まず文書の chunking と metadata 設計、その次に Azure AI Search での検索精度確認です。そのうえで Azure OpenAI をつなぎ、引用付き回答、fallback、ログ、token/latency の見える化を追加します。

### Q9. セキュリティ面では何を意識しましたか。

**A.**  
Day 1 時点では Azure 実キーを不要にし、環境変数契約だけを先に定義しました。本番を見据えるなら、フロントエンドから直接モデルを呼ばず、バックエンド経由にし、将来的には Entra ID や Managed Identity を使う前提で考えています。

### Q10. この取り組みを一言で言うと何ですか。

**A.**  
単なるデモではなく、Azure RAG へ発展可能な PoC 基盤を、設計・実装・文書化まで含めて形にした取り組みです。
